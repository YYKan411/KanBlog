## What this system does

A static, bilingual (Cantonese / English) personal-essay site. Posts are hand-authored HTML files in `posts/` (or composed via the web editor). On every push that touches `posts/`, GitHub Actions runs `scripts/build.js`, which regenerates the homepage post list, the sitemap, and the RSS feed; Cloudflare Pages then redeploys. No framework, no build step the author has to run by hand, no monthly cost.

The site has two faces: a masonry **homepage** that lists every post, and a per-post **article page** that renders bilingual prose side-by-side (desktop) or with a language toggle (mobile). *(2026-06-13: a third area was added вҖ” a **MiniGames** section at `/minigames/`, see its own section below.)*

## Build & deploy flow

```
Author commits to  posts/**  OR  scripts/build.js  OR  about.html
    вҶ’ GitHub Actions (build.yml)
        вҶ’ node scripts/build.js
            вҶ’ scans posts/*.html  (skips _TEMPLATE.html and any _*.html)
            вҶ’ extracts metadata from each post's <head>
            вҶ’ rewrites:
                В· app.js       (the SAMPLE_POSTS array)
                В· sitemap.xml  (posts + STATIC_PAGES: /minigames/, /minigames/catspuzzle/)
                В· feed.xml
                В· llms.txt     (GEO map вҖ” added 2026-06-13)
            вҶ’ commits the regenerated files as github-actions[bot]
              with message "...[skip ci]"  (so it does not re-trigger itself)
    вҶ’ Cloudflare Pages detects the commit вҶ’ redeploys
    вҶ’ live in ~1вҖ“2 minutes


Edits to  style.css / app.js / post.js / index.html / editor.html  etc.:
    вҶ’ build.js does NOT run (these are not in the watched paths)
    вҶ’ Cloudflare Pages STILL redeploys on the push (it watches every commit)
```

**Watched paths** (in `build.yml`): `posts/**`, `scripts/build.js`, `about.html`. Anything else deploys via Cloudflare but does not run the build script.

> вҡ пёҸ **`build.yml` itself is NOT a watched path.** Editing `build.yml` alone (e.g. to add a file to the bot's `git add`) does **not** re-run the Action. Commit it **together with a `build.js` change**, or make a trivial `build.js` edit afterward to re-trigger. *(This bit us on 2026-06-13: `build.js` + `build.yml` for `llms.txt` were committed separately, so `llms.txt` was generated in CI but never committed until a follow-up `build.js` edit re-ran the Action with the updated `build.yml`.)*

### The one coupling that can bite you

`build.js` locates the post list by a regular expression that matches `const SAMPLE_POSTS = [ вҖҰ ];` inside `app.js`, and rewrites that block. **Renaming the variable silently breaks auto-update** вҖ” the regex stops matching, `build.js` logs a warning and writes nothing, and new posts never appear on the homepage.

Cosmetic debt: the name says `SAMPLE_POSTS` but it is the *live* list, not a placeholder. Leave the name as-is unless you update the regex in `build.js` in the same change. (See Roadmap.)

## Post metadata contract

`build.js` reads these from each post's `<head>` to build the homepage card, sitemap entry, and RSS item:

| Source in the post | Used for |
|---|---|
| `<title>` (suffix " вҖ” иЁҖеҸҲеӢӨ" / "Kaga Chung" stripped) | Card title; sitemap |
| `meta[name="description"]` | Fallback for excerpt |
| `meta[name="excerpt"]` | Homepage card excerpt + RSS description |
| `meta[name="keywords"]` | Tags (filtered to the canonical set) |
| `meta[property="article:published_time"]` | Date вҖ” sort order, sitemap `lastmod`, RSS `pubDate` |
| `meta[property="og:image"]` | Card cover image |
| `meta[name="featured"]` = `true` | Renders as a dark "featured" card |

**Constraints to remember:**
- The meta-tag regex is **attribute-order-sensitive**: `name`/`property` must come *before* `content`. Tags written the other way round are not read.
- A post with **no title or no date is skipped** entirely.
- **`og:image` doubles as the card cover** вҖ” so `build.js` has a `DEFAULT_OG_IMAGE` guard (added 2026-06-13): if a post's `og:image` equals the generic default share image (`/images/og-default.png`), it is treated as *no cover* (text-only card), while still serving as the post's social thumbnail. `_TEMPLATE.html` bakes that default in, so future cover-less posts share with a thumbnail without polluting the home grid. A real cover (any other URL) still becomes the card image as before.

## Canonical tags

```
ж•Јж–Ү В· йҒҠиЁҳ В· е“ІжҖқ В· з§»иӢұ В· зү© В· дәә В· зӨҫжңғ В· йҰҷжёҜ
```

Defined in **two places that must stay in sync**: `build.js` (`extractTags`) and `editor.html` (`CANONICAL_TAGS`). Any tag outside this set is **silently dropped** by `build.js`. The editor surfaces a live warning and a confirm dialog before publishing an off-set tag.

## Tag routing (how filtering works)

The homepage reads the active tag from the **URL hash** (e.g. `yykan.uk/#ж•Јж–Ү`), **not** a query string. `app.js` decodes the hash and filters the post list; clicking a tag button sets the hash.

Post tag links must therefore point to `../index.html#<tag>`. Both encoded (`#%E6%95%A3%E6%96%87`) and plain (`#ж•Јж–Ү`) forms work, because `app.js` runs `decodeURIComponent`. *(A `?tag=` query string does nothing вҖ” this was the Candle Blossoms bug fixed 2026-06-06.)*

## Bilingual rendering (`post.js` + CSS)

- Prose is authored as `.pair` blocks: a `.zh` paragraph plus an optional `.en` paragraph.
- **Desktop:** `.zh` / `.en` sit side-by-side in a 50/50 grid. A pair with no `.en` collapses to a single column (via the CSS `:has()` selector).
- **Mobile (вүӨ720px):** Chinese only by default; `post.js` injects a 3-way segmented toggle (дёӯ / дёӯ + EN / EN) whenever the article contains any `.en`.
- **Verse mode:** add `class="verse"` to `.article-body` вҶ’ `white-space: pre-line` preserves a poem's line breaks (and full-width-space spacing within a line). *Added 2026-06-06; applied to the Ballad.*
- **Share buttons** (threads / facebook / x / linkedin / copy link) are injected by `post.js` after `.article-tags` on every post вҖ” old and new alike, no per-post markup needed.

### In-article image layouts (hand-coded; CSS already supports them)
- Default `<img>` вҖ” full column width.
- `<figure>` + `<figcaption>` вҖ” captioned image.
- `.img-row` вҖ” two images side-by-side (stacks on mobile).
- `.img-inset` вҖ” smaller centred image.

These exist in `style.css` and `_TEMPLATE.html` but the **editor cannot produce them** (see below).

## MiniGames (`/minigames/`) вҖ” added 2026-06-13

A small section listing browser games built for fun. Two layers:

- **Gallery hub вҖ” `minigames/index.html`:** a **standalone static page** (not processed by `build.js`). It reuses the blog's look by linking the live stylesheet with an **absolute** path (`<link href="/style.css">`) and the same fonts, and replicates the masthead + footer. The "е°ҸйҒҠжҲІ В· Mini Games" label sits as **small text under the site identity** in the masthead. Game cards are rendered by an **inline `<script>`** holding a `GAMES` array, using the blog's own card classes (`.card`, `.card-cover`, `.card-body`, `.card-title`, `.card-excerpt`, `.card-meta`, `.card-tag`) вҖ” so they look identical to post cards.
- **The "Cats" game вҖ” `minigames/catspuzzle/`:** a Star-Battle / Queens-style logic puzzle (one cat per colour/row/column, no two cats touch incl. diagonally). Built with **React + TypeScript + Tailwind 4 + Vite**, fully client-side, themed in Library Linen. Has a "вҶҗ Mini Games" link (вҶ’ `/minigames/`) and a footer link home. **Only the built output is in the repo** (`index.html` + `assets/index-<hash>.js/.css`) вҖ” *the React source is not in the repo.*

### вһ• How to add a new game
1. Build/deploy the game into `minigames/<game>/`.
2. Add a cover to `images/`.
3. Append one entry to the `GAMES` array in `minigames/index.html` (`{ title, excerpt, cover, url, tags, meta }`).
4. Add it to `STATIC_PAGES` in `scripts/build.js` (sitemap) and to the `## Mini Games` list in the `llms.txt` generator (also in `build.js`).

### вҡ пёҸ Vite asset-hash gotcha
Vite re-hashes `index-<hash>.js/.css` on every build, so **updating the game = upload the new asset files AND delete the old hashed ones** (orphans are harmless but messy). Current live assets: `index-BTKfVq4T.js` / `index-DDfAfMFm.css`. *Future improvement (Roadmap): configure Vite for stable filenames so updates are a plain overwrite.* The game is built with Vite `base: './'` so relative asset paths work from any subfolder.

## The editor (`editor.html` + Worker)

- Single-page composer; `noindex` and disallowed in `robots.txt`.
- **Auth:** GitHub OAuth вҶ’ the Worker (`kanblog-editor-api`) вҶ’ a session token kept in `localStorage`. The `/auth/start` and `/auth/callback` routes are forwarded to the Worker by `_redirects`.
- Builds the full post HTML client-side (SEO meta, OG, Twitter card, BlogPosting JSON-LD) and POSTs it to the Worker's `/api/commit-file`, which commits to the repo. The commit then flows through the normal build/deploy pipeline.
- **Body composer supports `.pair` blocks only** (Chinese + optional English), plus a single cover-image field with alt text.

**Known limits (relevant to illustrated "дёҠж–ҮдёҠең–" posts):**
- Publish-only вҖ” it cannot edit an existing post (edit via GitHub directly).
- No inline images, figures, blockquotes, headings, or verse. Any post needing these is hand-coded from `_TEMPLATE.html`.
- *Highest-leverage upgrade:* an "image block" in the composer (src + alt + caption + layout) emitting the HTML the CSS already supports. (See Roadmap.)

## Domain, redirects, Cloudflare

- **Custom domains:** `yykan.uk` (apex, canonical) and `www.yykan.uk` вҖ” both active, SSL active. **`www` 301s to the apex** via a Cloudflare **Redirect Rule** named `www to apex` (zone `yykan.uk` вҶ’ Rules вҶ’ Redirect Rules; `http.host eq "www.yykan.uk"` вҶ’ dynamic 301 вҶ’ `concat("https://yykan.uk", http.request.uri.path)`, preserve query string). Done & verified 2026-06-14 (path + query preserved, no loop). The `www.yykan.uk/* вҶ’ yykan.uk` line in `_redirects` is a **dormant fallback only** вҖ” Cloudflare Pages `_redirects` matches path, not hostname, so it never fires; the Redirect Rule is what does the work (same pattern as the pages.dev Bulk Redirect below).
- **Old domain:** `kanblog.pages.dev` вҶ’ `yykan.uk` via a Cloudflare **Bulk Redirect** (account-level rule, subpath + query + path-suffix preserved) and the `_redirects` 301.
- **Pages project:** `kanblog` (the source of the old `*.pages.dev` URL). Auto-deploys on every GitHub push.
- **Security headers (`_headers`, 2026-06-14):** Cloudflare Pages serves the repo-root `_headers` file on every response. Sets `Strict-Transport-Security` (1y, `includeSubDomains; preload`), a **tailored CSP** (allows Google Fonts `style-src`/`font-src`, inline JSON-LD via `script-src 'unsafe-inline'`, and the editor's Worker via `connect-src https://kanblog-editor-api.7r6r7hf474.workers.dev`), `X-Frame-Options: DENY`, `Permissions-Policy`, plus a 1-year `Cache-Control` on `/images/*` (previously 4h). вҡ пёҸ **CSP note:** the `connect-src` Worker allowance is load-bearing for the editor вҖ” without it `editor.html`'s commit/OAuth `fetch()` would be blocked. If the editor ever stops committing with a console `Refused to connect`, check this directive first. Editor login + commit round-trip **verified working under the CSP (2026-06-14)**.

## Known issues & design notes

- **`SAMPLE_POSTS` naming debt** вҖ” the live list is misleadingly named (see "The one coupling that can bite you").
- **Masonry reading order** вҖ” CSS columns fill top-to-bottom *per column*, so the visual order does not strictly match newest-first left-to-right. Accepted trade-off for zero JS.
- **Card cover CLS вҖ” accepted by design** вҖ” the masonry deliberately lets each card keep its natural image height (Pinterest-style freeflow), so card covers carry no fixed dimensions and can shift slightly as they load. This is an intentional aesthetic choice, not a defect. The proper "freeflow *and* no CLS" fix (reserving each card's natural-ratio space from known image dimensions) is on the Roadmap.
- **Article hero CLS вҖ” open** вҖ” the full-width `.article-cover` on each post has no `width`/`height`, so content can shift once it loads. Fixable *without* cropping by adding each image's intrinsic dimensions (needs the four cover sizes). Not yet done.
- **Font weight** вҖ” Noto Serif HK loads four weights (400/500/600/700); `700` is unused. Note: browsers do not download unused-weight font files, so dropping `700` trims only a little font CSS, not real bytes. A meaningful payload cut means dropping a *used* weight (e.g. `400 + 600`, losing `500`) вҖ” a visual trade-off, on the Roadmap.
- **Editor** вҖ” publish-only, pairs-only (see editor section). *Open.*
- **build.js meta regex** вҖ” attribute-order-sensitive (`name`/`property` before `content`).

- **About page вҖ” two `<h1>`s** (should be one); also no `twitter:card`. *Open (2026-06-13).*
- **Homepage / MiniGames cards are JS-injected** вҖ” no static `<a href>` links in the HTML; Google renders JS so it indexes fine, but a `<noscript>` link list would harden it for non-JS / AI crawlers. *Open (2026-06-13).*

*Resolved 2026-06-06:* meta-text contrast (`--ink-faded` вҶ’ `#75603e`), `prefers-reduced-motion`, `:focus-visible`, site favicon, homepage `og:image`, and the legacy-name strip in `readTitle`.

*Resolved 2026-06-13:* **soft-404** (Cloudflare returned 200 + homepage for any missing URL вҶ’ fixed with a branded `404.html`, now real 404s); Person `sameAs`; raster OG images (SVG didn't render in social scrapers); `/minigames/` + game in the sitemap; auto-generated `llms.txt`; game-page canonical/OG/Twitter meta; default OG for About + future cover-less posts.
