## What this system does

A static, bilingual (Cantonese / English) personal-essay site. Posts are hand-authored HTML files in `posts/` (or composed via the web editor). On every push that touches `posts/`, GitHub Actions runs `scripts/build.js`, which regenerates the homepage post list, the sitemap, and the RSS feed; Cloudflare Pages then redeploys. No framework, no build step the author has to run by hand, no monthly cost.

The site has four faces: a masonry **homepage** that lists every post; a per-post **article page** that renders bilingual prose side-by-side (desktop) or with a language toggle (mobile); a **MiniGames** shelf at `/minigames/` (2026-06-13); and a **Learning** shelf at `/learning/` (2026-07-20) — practice notes and problems, starting with SQL. Site chrome (masthead + footer) is shared; footer order is `主頁/home · 關於/about · 小遊戲/mini games · 學習/learning · rss`. **Chrome bilingual glue:** within one control use `/` (e.g. `主頁/home`); use ` · ` only *between* footer items — so middots read as separators, not as part of each button (refined 2026-07-21).

## Build & deploy flow

```
Author commits to  posts/**  OR  scripts/build.js  OR  about.html
    → GitHub Actions (build.yml)
        → node scripts/build.js
            → scans posts/*.html  (skips _TEMPLATE.html and any _*.html)
            → extracts metadata from each post's <head>
            → rewrites:
                · app.js       (the POSTS array)
                · sitemap.xml  (posts + STATIC_PAGES: /minigames/*, /learning/, /learning/sql/)
                · feed.xml
                · llms.txt     (GEO map — Mini Games + Learning sections)
                · index.html   (<noscript> post links between BUILD:POSTS markers — added 2026-06-14)
                · post.js      (POSTS_NAV prev/next data between BUILD:NAV markers — added 2026-06-15)
            → commits the regenerated files as github-actions[bot]
              with message "...[skip ci]"  (so it does not re-trigger itself)
    → Cloudflare Pages detects the commit → redeploys
    → live in ~1–2 minutes


Edits to  style.css / app.js / post.js / index.html / editor.html  etc.:
    → build.js does NOT run (these are not in the watched paths)
    → Cloudflare Pages STILL redeploys on the push (it watches every commit)
```

**Watched paths** (in `build.yml`): `posts/**`, `scripts/build.js`, `about.html`. Anything else deploys via Cloudflare but does not run the build script.

> ⚠️ **`build.yml` itself is NOT a watched path.** Editing `build.yml` alone (e.g. to add a file to the bot's `git add`) does **not** re-run the Action. Commit it **together with a `build.js` change**, or make a trivial `build.js` edit afterward to re-trigger. *(This bit us on 2026-06-13: `build.js` + `build.yml` for `llms.txt` were committed separately, so `llms.txt` was generated in CI but never committed until a follow-up `build.js` edit re-ran the Action with the updated `build.yml`.)*

### The one coupling that can bite you

`build.js` locates the post list by a regular expression that matches `const POSTS = [ … ];` inside `app.js`, and rewrites that block. **Renaming the variable silently breaks auto-update** — the regex stops matching, `build.js` logs a warning and writes nothing, and new posts never appear on the homepage.

The live list is named `POSTS` (renamed from the old `SAMPLE_POSTS` placeholder). Keep the `build.js` regex in sync if you ever rename it again.

## Post metadata contract

`build.js` reads these from each post's `<head>` to build the homepage card, sitemap entry, and RSS item:

| Source in the post | Used for |
|---|---|
| `<title>` (suffix " — 言又勤" stripped) | Card title; sitemap |
| `meta[name="description"]` | Fallback for excerpt |
| `meta[name="excerpt"]` | Homepage card excerpt + RSS description |
| `meta[name="tags"]` | Homepage nav taxonomy (canonical set only). Prefer this. |
| `meta[name="keywords"]` | SEO extras OK here; also a **fallback** for tags if `tags` is absent (filtered to the canonical set) |
| `meta[property="article:published_time"]` | Date — sort order, sitemap `lastmod`, RSS `pubDate` |
| `meta[property="og:image"]` | Card cover image |
| `meta[name="featured"]` = `true` | Renders as a dark "featured" card |

**Constraints to remember:**
- `readMeta` accepts `name`/`property` before *or* after `content` (relaxed 2026-07-21). Still prefer the conventional order in new posts.
- A post with **no title or no date is skipped** entirely.
- **`og:image` doubles as the card cover** — so `build.js` has a `DEFAULT_OG_IMAGE` guard (added 2026-06-13): if a post's `og:image` equals the generic default share image (`/images/og-default.png`), it is treated as *no cover* (text-only card), while still serving as the post's social thumbnail. `_TEMPLATE.html` bakes that default in, so future cover-less posts share with a thumbnail without polluting the home grid. A real cover (any other URL) still becomes the card image as before.

**🖼️ Image format policy (set 2026-06-14) — read before adding a post with a cover:**
- **Display images are WebP; `og:image` stays a raster (JPEG/PNG).** WebP is ~30–40% smaller and is what visitors load (home card + in-article hero). But several social scrapers (Facebook/WhatsApp) don't render WebP thumbnails, so `og:image` / `twitter:image` / the JSON-LD `image` keep pointing at a `.jpg`/`.png`.
- **Per cover you therefore keep two files:** `images/<name>.webp` (display) **and** `images/<name>.jpg` (social/og). The in-article `<img class="article-cover">` uses `../images/<name>.webp`; the meta tags use `https://yykan.uk/images/<name>.jpg`.
- **`build.js` bridges them:** it reads `og:image` (the `.jpg`) as before, then — if a `.webp` sibling exists in `images/` — swaps the extension so the **home-card cover (`POSTS[].cover`) is the WebP**. No `.webp` sibling → it falls back to the original raster. So the home grid is WebP, social stays JPEG, automatically.
- **Converting:** `cwebp -q 80 -m 6 -metadata none in.jpg -o out.webp` (q80 is the safe quality/size balance for these photographic covers; lower q saves more but risks visible artefacts on a design-led site). The original JPEG is **kept**, not deleted. (On *not* deleting old JPEGs to "save space": deleting from the tree does not shrink `.git` history anyway — see the storage reasoning; the policy is to keep one JPEG per cover for `og:image` regardless.)
- **At scale:** because we never accumulate orphan JPEGs (each JPEG has a job: `og:image`), there is no future "mass-delete" cleanup. If the image corpus ever gets genuinely large, the move is to host images off-repo (Cloudflare R2) rather than prune history.

## Canonical tags

```
散文 · 遊記 · 哲思 · 移英 · 物 · 人 · 社會 · 香港
```

Defined in **two places that must stay in sync**: `build.js` (`CANONICAL_TAGS` / `extractTags`) and `editor.html` (`CANONICAL_TAGS`). Posts should declare nav tags in `<meta name="tags">`; `keywords` may hold SEO extras and remains a fallback if `tags` is missing. Any token outside the canonical set is **silently dropped** by `build.js`. The editor surfaces a live warning and a confirm dialog before publishing an off-set tag.

---

## Tag routing (how filtering works)

The homepage reads the active tag from the **URL hash** (e.g. `yykan.uk/#散文`), **not** a query string. `app.js` decodes the hash and filters the post list; clicking a tag button sets the hash.

Post tag links must therefore point to `../index.html#<tag>`. Both encoded (`#%E6%95%A3%E6%96%87`) and plain (`#散文`) forms work, because `app.js` runs `decodeURIComponent`. *(A `?tag=` query string does nothing — this was the Candle Blossoms bug fixed 2026-06-06.)*

## Bilingual rendering (`post.js` + CSS)

- Prose is authored as `.pair` blocks: a `.zh` paragraph plus an optional `.en` paragraph.
- **Desktop:** `.zh` / `.en` sit side-by-side in a 50/50 grid. A pair with no `.en` collapses to a single column (via the CSS `:has()` selector).
- **Mobile (≤720px):** Chinese only by default; `post.js` injects a 3-way segmented toggle (中 / 中 + EN / EN) whenever the article contains any `.en`.
- **Verse mode:** add `class="verse"` to `.article-body` → `white-space: pre-line` preserves a poem's line breaks (and full-width-space spacing within a line). *Added 2026-06-06; applied to the Ballad.*
- **Share buttons** (threads / facebook / x / linkedin / copy link) are injected by `post.js` after `.article-tags` on every post — old and new alike, no per-post markup needed.
- **Previous / next nav** (added 2026-06-15) — `post.js` injects chronological neighbour links (older left, newer right) after the share buttons. It reads `POSTS_NAV`, an ordered (newest-first) array of `{slug, title, url}` that `build.js` regenerates between `/* BUILD:NAV_START/END */` markers at the top of `post.js`. Same single-source pattern as the homepage `<noscript>` list: one build step covers every post, no per-post markup, no backfill. Current post is matched by slug from the URL path; first/last posts show only the one neighbour that exists. `build.yml` now commits `post.js` alongside the other generated files.

### In-article image layouts (hand-coded; CSS already supports them)
- Default `<img>` — full column width.
- `<figure>` + `<figcaption>` — captioned image.
- `.img-row` — two images side-by-side (stacks on mobile).
- `.img-inset` — smaller centred image.

These exist in `style.css` and `_TEMPLATE.html` but the **editor cannot produce them** (see below).

### 🎮 The Game Contract (added 2026-06-15)

Games are **apps, not posts**: each game's *interior* (gameplay UI, board, controls,
layout, animation) should differ — **do not template it.** But every game shares an
*outer contract* so it belongs to the site and is discoverable. **Standardise the
shell; keep the interior free.** (Same philosophy as posts having a rigid
`_TEMPLATE.html` head but free body content — just inverted: here the body is free and
only the shell is fixed.)

**FREE — the interior.** Everything about how the game looks and plays. A puzzle, an
arcade game, a word game should look nothing alike. Reference the existing game(s) for
*tone* (quiet, paper, Library Linen restraint), **not** for layout.

**FIXED — the contract every game must honour:**

1. **Frame & tone** — use the Library Linen tokens (below) + the three site fonts;
   restraint over flash. Provide a `← Mini Games` link (→ `/minigames/`) and a link
   home (→ `/`). The game must visually read as part of yykan.uk.
2. **Page meta** — `minigames/<game>/index.html` carries the head block below
   (title / description / canonical / robots / OG / Twitter / JSON-LD). Same discipline
   as `_TEMPLATE.html` for posts.
3. **Registration — 3 synchronized edits (the easy-to-forget part):**
   - `minigames/index.html` → append to the `GAMES` array: `{ title, excerpt, cover, url, tags, meta }`
   - `scripts/build.js` → add to `STATIC_PAGES` (→ sitemap): `{ file, loc, priority }`
   - `scripts/build.js` → add a line to the `## Mini Games` list in the `llms.txt` generator
4. **Share image** — a 1200×630 **PNG** `images/og-<game>.png` (raster; SVG doesn't
   render in social scrapers). Until it exists, the page falls back to `og-minigames.png`.
5. **Hub presentation — one shelf, three promises.** Every card follows the same
   editorial rhythm even though each game keeps its own identity:
   - **Title:** always bilingual, `中文名 · English Name`.
   - **Excerpt:** one Chinese sentence followed by one English sentence. Speak from
     the player's point of view: **what they do + what they take away**. Do not dump
     rules, question counts, engine names, frameworks or other implementation detail.
     The English line may complement the Chinese; it must not mechanically translate it.
   - **Tags:** exactly two at launch — one Chinese and one English, at the same level
     of specificity (for example `象棋` + `xiangqi`, not `象棋` + `board`).
   - **Cover:** a text-free **3:2 WebP atmosphere image**, not a raw UI screenshot.
     Keep the Library Linen restraint (warm paper, muted colour, editorial light) and
     include a trace of human presence — a pencil, glass, hand, unfinished move. Covers
     should feel like one photographic series while each game's objects remain distinct.
     The `GAMES` entry must carry a descriptive `alt`; do not reuse the title as alt text.
   - **Separation of jobs:** hub cover creates mood; `og-<game>.png` creates a legible
     social preview. They need not be the same composition.
6. **Asset hygiene** — Vite `base: './'` (relative paths, works from the subfolder).
   Prefer **stable output filenames** so updating a game is a plain overwrite; if
   filenames stay hashed, deleting the old hashed files on update is mandatory.

### ⚠️ CSP + sql.js (load-bearing)

Site CSP is **`'self'`-only** for scripts and `connect-src` (plus the editor Worker). Fetching `sql.js` from a CDN **fails** (`Failed to fetch dynamically imported module`). Therefore:

1. **Vendor** `sql-wasm.js` + `sql-wasm.wasm` under `learning/sql/vendor/` and **commit them**.
2. Load with a **classic** `<script src="./vendor/sql-wasm.js">` before the ES module `app.js` (exposes `globalThis.initSqlJs`).
3. `db.js` calls `initSqlJs({ locateFile: … })` pointing at `./vendor/`.
4. CSP already allows `'wasm-unsafe-eval'` under `script-src` (needed for sql.js WASM).
