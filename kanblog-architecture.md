---

## What this system does

A static, bilingual (Cantonese / English) personal-essay site. Posts are hand-authored HTML files in `posts/` (or composed via the web editor). On every push that touches `posts/`, GitHub Actions runs `scripts/build.js`, which regenerates the homepage post list, the sitemap, and the RSS feed; Cloudflare Pages then redeploys. No framework, no build step the author has to run by hand, no monthly cost.

The site has two faces: a masonry **homepage** that lists every post, and a per-post **article page** that renders bilingual prose side-by-side (desktop) or with a language toggle (mobile).

## Build & deploy flow

```
Author commits to  posts/**  OR  scripts/build.js  OR  about.html
    → GitHub Actions (build.yml)
        → node scripts/build.js
            → scans posts/*.html  (skips _TEMPLATE.html and any _*.html)
            → extracts metadata from each post's <head>
            → rewrites:
                · app.js       (the SAMPLE_POSTS array)
                · sitemap.xml
                · feed.xml
            → commits the regenerated files as github-actions[bot]
              with message "...[skip ci]"  (so it does not re-trigger itself)
    → Cloudflare Pages detects the commit → redeploys
    → live in ~1–2 minutes


Edits to  style.css / app.js / post.js / index.html / editor.html  etc.:
    → build.js does NOT run (these are not in the watched paths)
    → Cloudflare Pages STILL redeploys on the push (it watches every commit)
```

**Watched paths** (in `build.yml`): `posts/**`, `scripts/build.js`, `about.html`. Anything else deploys via Cloudflare but does not run the build script.

### The one coupling that can bite you

`build.js` locates the post list by a regular expression that matches `const SAMPLE_POSTS = [ … ];` inside `app.js`, and rewrites that block. **Renaming the variable silently breaks auto-update** — the regex stops matching, `build.js` logs a warning and writes nothing, and new posts never appear on the homepage.

Cosmetic debt: the name says `SAMPLE_POSTS` but it is the *live* list, not a placeholder. Leave the name as-is unless you update the regex in `build.js` in the same change. (See Roadmap.)

## Post metadata contract

`build.js` reads these from each post's `<head>` to build the homepage card, sitemap entry, and RSS item:

| Source in the post | Used for |
|---|---|
| `<title>` (suffix " — 言又勤" / "Kaga Chung" stripped) | Card title; sitemap |
| `meta[name="description"]` | Fallback for excerpt |
| `meta[name="excerpt"]` | Homepage card excerpt + RSS description |
| `meta[name="keywords"]` | Tags (filtered to the canonical set) |
| `meta[property="article:published_time"]` | Date — sort order, sitemap `lastmod`, RSS `pubDate` |
| `meta[property="og:image"]` | Card cover image |
| `meta[name="featured"]` = `true` | Renders as a dark "featured" card |

**Two constraints to remember:**
- The meta-tag regex is **attribute-order-sensitive**: `name`/`property` must come *before* `content`. Tags written the other way round are not read.
- A post with **no title or no date is skipped** entirely.

## Canonical tags

```
散文 · 遊記 · 哲思 · 移英 · 物 · 人 · 社會 · 香港
```

Defined in **two places that must stay in sync**: `build.js` (`extractTags`) and `editor.html` (`CANONICAL_TAGS`). Any tag outside this set is **silently dropped** by `build.js`. The editor surfaces a live warning and a confirm dialog before publishing an off-set tag.

---

## Tag routing (how filtering works)

The homepage reads the active tag from the **URL hash** (e.g. `yykan.uk/#散文`), **not** a query string. `app.js` decodes the hash and filters the post list; clicking a tag button sets the hash.

Post tag links must therefore point to `../index.html#<tag>`. Both encoded (`#%E6%95%A3%E6%96%87`) and plain (`#散文`) forms work, because `app.js` runs `decodeURIComponent`. *(A `?tag=` query string does nothing — this was the Candle Blossoms bug fixed 2026-06-06.)*

## Bilingual rendering (`post.js` + CSS)

- Prose is authored as `.pair` blocks: a `.zh` paragraph plus an optional `.en` paragraph.
- **Desktop:** `.zh` / `.en` sit side-by-side in a 50/50 grid. A pair with no `.en` collapses to a single column (via the CSS `:has()` selector).
- **Mobile (≤720px):** Chinese only by default; `post.js` injects a 3-way segmented toggle (中 / 中 + EN / EN) whenever the article contains any `.en`.
- **Verse mode:** add `class="verse"` to `.article-body` → `white-space: pre-line` preserves a poem's line breaks (and full-width-space spacing within a line). *Added 2026-06-06; applied to the Ballad.*
- **Share buttons** (threads / facebook / x / linkedin / copy link) are injected by `post.js` after `.article-tags` on every post — old and new alike, no per-post markup needed.

### In-article image layouts (hand-coded; CSS already supports them)
- Default `<img>` — full column width.
- `<figure>` + `<figcaption>` — captioned image.
- `.img-row` — two images side-by-side (stacks on mobile).
- `.img-inset` — smaller centred image.

These exist in `style.css` and `_TEMPLATE.html` but the **editor cannot produce them** (see below).

## The editor (`editor.html` + Worker)

- Single-page composer; `noindex` and disallowed in `robots.txt`.
- **Auth:** GitHub OAuth → the Worker (`kanblog-editor-api`) → a session token kept in `localStorage`. The `/auth/start` and `/auth/callback` routes are forwarded to the Worker by `_redirects`.
- Builds the full post HTML client-side (SEO meta, OG, Twitter card, BlogPosting JSON-LD) and POSTs it to the Worker's `/api/commit-file`, which commits to the repo. The commit then flows through the normal build/deploy pipeline.
- **Body composer supports `.pair` blocks only** (Chinese + optional English), plus a single cover-image field with alt text.

**Known limits (relevant to illustrated "上文上圖" posts):**
- Publish-only — it cannot edit an existing post (edit via GitHub directly).
- No inline images, figures, blockquotes, headings, or verse. Any post needing these is hand-coded from `_TEMPLATE.html`.
- *Highest-leverage upgrade:* an "image block" in the composer (src + alt + caption + layout) emitting the HTML the CSS already supports. (See Roadmap.)

## SEO / AEO setup

- **Per post:** title, description, keywords, canonical, robots, Open Graph, Twitter card, and BlogPosting JSON-LD.
- **Site-wide:** WebSite JSON-LD on the homepage; AboutPage + Person JSON-LD on the About page; auto-generated `sitemap.xml`, `feed.xml`, `robots.txt`; `google-site-verification` on the homepage.
- **Search Console:** `yykan.uk` is a DNS-verified domain property. The old `kanblog.pages.dev` property is retained to watch the 301 transition.
- **Open gaps** (see Roadmap): the Person entity has no `sameAs` / `jobTitle` / `knowsAbout`; the homepage has no `og:image`; there is no favicon.

## Design system — "Library Linen"

- **Fonts:** Cormorant Garamond (display) · Noto Serif HK (body) · JetBrains Mono (meta).
- **Palette** (the `:root` block in `style.css`): `--ink #3d2f1f` · `--paper #ebe5d4` · `--accent #3a5a3a` (墨綠, used sparingly) · `--ink-soft #6a5840` · `--ink-faded #998566` · `--rule #c9bfa3`.
- **Masonry:** pure CSS columns — no JS library. Cards fade in with a staggered delay.
- **To restyle:** edit colours / fonts / sizes at the `:root` block; homepage title + tagline in `index.html`; About copy in `about.html`.

## Known issues & design notes

- **`SAMPLE_POSTS` naming debt** — the live list is misleadingly named (see "The one coupling that can bite you").
- **Masonry reading order** — CSS columns fill top-to-bottom *per column*, so the visual order does not strictly match newest-first left-to-right. Accepted trade-off for zero JS.
- **Card cover CLS — accepted by design** — the masonry deliberately lets each card keep its natural image height (Pinterest-style freeflow), so card covers carry no fixed dimensions and can shift slightly as they load. This is an intentional aesthetic choice, not a defect. The proper "freeflow *and* no CLS" fix (reserving each card's natural-ratio space from known image dimensions) is on the Roadmap.
- **Article hero CLS — open** — the full-width `.article-cover` on each post has no `width`/`height`, so content can shift once it loads. Fixable *without* cropping by adding each image's intrinsic dimensions (needs the four cover sizes). Not yet done.
- **Font weight** — Noto Serif HK loads four weights (400/500/600/700); `700` is unused. Note: browsers do not download unused-weight font files, so dropping `700` trims only a little font CSS, not real bytes. A meaningful payload cut means dropping a *used* weight (e.g. `400 + 600`, losing `500`) — a visual trade-off, on the Roadmap.
- **Editor** — publish-only, pairs-only (see editor section). *Open.*
- **build.js meta regex** — attribute-order-sensitive (`name`/`property` before `content`).

*Resolved 2026-06-06:* meta-text contrast (`--ink-faded` → `#75603e`), `prefers-reduced-motion`, `:focus-visible`, site favicon, homepage `og:image`, and the legacy-name strip in `readTitle`.

2. **Homepage** `yykan.uk` → loads; all posts render; tag bar present.
3. **Console** F12 → no warnings (in particular, no `Using sample posts:`).

6. **Verse** → open the Ballad → four stanzas render with line breaks intact.
7. **Pipeline** GitHub Actions → last build is green; a bot commit `chore: auto-update…` appears after a post change.

9. **Editor** `yykan.uk/editor.html` → GitHub sign-in works; an off-canonical tag triggers the warning.

## Roadmap / pick-up

**Performance / a11y (reviewed and deferred in the 2026-06-06 quick-win pass):**
- *Freeflow + no CLS for cards* — keep the Pinterest-style natural card heights but eliminate layout shift by reserving each card's natural-ratio space. Mechanism: store each cover's intrinsic dimensions in post metadata → read in `build.js` → apply as `width`/`height` in `app.js`'s `makeCard`. Multi-file; this is the *non-cropping* version of the card-CLS fix (the 3:2 crop option was rejected on purpose — uniform covers kill the freeflow look).
- *Article hero dimensions* — add intrinsic `width`/`height` to each `.article-cover` (no cropping, no uniformity). Quick once the four cover sizes are to hand.
- *Font payload* — only worthwhile if dropping a *used* weight (e.g. `400 + 600`); a visual decision, not free. Dropping the unused `700` alone is near-zero benefit.

**Features:**
- Previous / next ("read next") links at the foot of each post — retention + internal-link SEO.
- Web Share API as the primary mobile share action (current links as fallback).
- Dark mode (`prefers-color-scheme`) — fits the candlelight theme.
- Enrich the Person entity: `sameAs` (IG / FB / Threads), `jobTitle`, `knowsAbout`, `alumniOf` — strengthens AEO / entity recognition.
- Client-side search over the post list — when the corpus grows.
- Cloudflare Web Analytics — free, cookie-less.
- Date-indexed archive page — when the corpus grows.

**Editor:**
- Add an inline image / figure block (highest leverage for illustrated posts).
- Allow editing existing posts, not just creating new ones.

**Tech debt:**
- Rename `SAMPLE_POSTS` → `POSTS` — **must update `build.js`'s regex in the same change**.
