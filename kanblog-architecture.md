## What this system does

A static, bilingual (Cantonese / English) personal-essay site. Posts are hand-authored HTML files in `posts/` (or composed via the web editor). On every push that touches `posts/`, GitHub Actions runs `scripts/build.js`, which regenerates the homepage post list, the sitemap, and the RSS feed; Cloudflare Pages then redeploys. No framework, no build step the author has to run by hand, no monthly cost.

The site has two faces: a masonry **homepage** that lists every post, and a per-post **article page** that renders bilingual prose side-by-side (desktop) or with a language toggle (mobile). *(2026-06-13: a third area was added — a **MiniGames** section at `/minigames/`, see its own section below.)*

## Build & deploy flow

```
Author commits to  posts/**  OR  scripts/build.js  OR  about.html
    → GitHub Actions (build.yml)
        → node scripts/build.js
            → scans posts/*.html  (skips _TEMPLATE.html and any _*.html)
            → extracts metadata from each post's <head>
            → rewrites:
                · app.js       (the SAMPLE_POSTS array)
                · sitemap.xml  (posts + STATIC_PAGES: /minigames/, /minigames/catspuzzle/)
                · feed.xml
                · llms.txt     (GEO map — added 2026-06-13)
                · index.html   (<noscript> post links between BUILD:POSTS markers — added 2026-06-14)
                · post.js      (POSTS_NAV prev/next data between BUILD:NAV markers — added 2026-06-15)
            → commits the regenerated files as github-actions[bot]
              with message "...[skip ci]"  (so it does not re-trigger itself)
    → Cloudflare Pages detects the commit → redeploys
    → live in ~1–2 minutes


Edits to  style.css / app.js / post.js / index.html / editor.html  etc.:
    → build.js does NOT run (these are not in the watched paths)
    → Cloudflare Pages STILL redeploys on the push (it watches every commit)
```

**Watched paths** (in `build.yml`): `posts/**`, `scripts/build.js`, `about.html`. Anything else deploys via Cloudflare but does not run the build script.

> ⚠️ **`build.yml` itself is NOT a watched path.** Editing `build.yml` alone (e.g. to add a file to the bot's `git add`) does **not** re-run the Action. Commit it **together with a `build.js` change**, or make a trivial `build.js` edit afterward to re-trigger. *(This bit us on 2026-06-13: `build.js` + `build.yml` for `llms.txt` were committed separately, so `llms.txt` was generated in CI but never committed until a follow-up `build.js` edit re-ran the Action with the updated `build.yml`.)*

### The one coupling that can bite you

`build.js` locates the post list by a regular expression that matches `const SAMPLE_POSTS = [ … ];` inside `app.js`, and rewrites that block. **Renaming the variable silently breaks auto-update** — the regex stops matching, `build.js` logs a warning and writes nothing, and new posts never appear on the homepage.

Cosmetic debt: the name says `SAMPLE_POSTS` but it is the *live* list, not a placeholder. Leave the name as-is unless you update the regex in `build.js` in the same change. (See Roadmap.)

## Post metadata contract

`build.js` reads these from each post's `<head>` to build the homepage card, sitemap entry, and RSS item:

| Source in the post | Used for |
|---|---|
| `<title>` (suffix " — 言又勤" / "Kaga Chung" stripped) | Card title; sitemap |
| `meta[name="description"]` | Fallback for excerpt |
| `meta[name="excerpt"]` | Homepage card excerpt + RSS description |
| `meta[name="keywords"]` | Tags (filtered to the canonical set) |
| `meta[property="article:published_time"]` | Date — sort order, sitemap `lastmod`, RSS `pubDate` |
| `meta[property="og:image"]` | Card cover image |
| `meta[name="featured"]` = `true` | Renders as a dark "featured" card |

**Constraints to remember:**
- The meta-tag regex is **attribute-order-sensitive**: `name`/`property` must come *before* `content`. Tags written the other way round are not read.
- A post with **no title or no date is skipped** entirely.
- **`og:image` doubles as the card cover** — so `build.js` has a `DEFAULT_OG_IMAGE` guard (added 2026-06-13): if a post's `og:image` equals the generic default share image (`/images/og-default.png`), it is treated as *no cover* (text-only card), while still serving as the post's social thumbnail. `_TEMPLATE.html` bakes that default in, so future cover-less posts share with a thumbnail without polluting the home grid. A real cover (any other URL) still becomes the card image as before.

**🖼️ Image format policy (set 2026-06-14) — read before adding a post with a cover:**
- **Display images are WebP; `og:image` stays a raster (JPEG/PNG).** WebP is ~30–40% smaller and is what visitors load (home card + in-article hero). But several social scrapers (Facebook/WhatsApp) don't render WebP thumbnails, so `og:image` / `twitter:image` / the JSON-LD `image` keep pointing at a `.jpg`/`.png`.
- **Per cover you therefore keep two files:** `images/<name>.webp` (display) **and** `images/<name>.jpg` (social/og). The in-article `<img class="article-cover">` uses `../images/<name>.webp`; the meta tags use `https://yykan.uk/images/<name>.jpg`.
- **`build.js` bridges them:** it reads `og:image` (the `.jpg`) as before, then — if a `.webp` sibling exists in `images/` — swaps the extension so the **home-card cover (`SAMPLE_POSTS[].cover`) is the WebP**. No `.webp` sibling → it falls back to the original raster. So the home grid is WebP, social stays JPEG, automatically.
- **Converting:** `cwebp -q 80 -m 6 -metadata none in.jpg -o out.webp` (q80 is the safe quality/size balance for these photographic covers; lower q saves more but risks visible artefacts on a design-led site). The original JPEG is **kept**, not deleted. (On *not* deleting old JPEGs to "save space": deleting from the tree does not shrink `.git` history anyway — see the storage reasoning; the policy is to keep one JPEG per cover for `og:image` regardless.)
- **At scale:** because we never accumulate orphan JPEGs (each JPEG has a job: `og:image`), there is no future "mass-delete" cleanup. If the image corpus ever gets genuinely large, the move is to host images off-repo (Cloudflare R2) rather than prune history.

## Canonical tags

```
散文 · 遊記 · 哲思 · 移英 · 物 · 人 · 社會 · 香港
```

Defined in **two places that must stay in sync**: `build.js` (`extractTags`) and `editor.html` (`CANONICAL_TAGS`). Any tag outside this set is **silently dropped** by `build.js`. The editor surfaces a live warning and a confirm dialog before publishing an off-set tag.

---

## Tag routing (how filtering works)

The homepage reads the active tag from the **URL hash** (e.g. `yykan.uk/#散文`), **not** a query string. `app.js` decodes the hash and filters the post list; clicking a tag button sets the hash.

Post tag links must therefore point to `../index.html#<tag>`. Both encoded (`#%E6%95%A3%E6%96%87`) and plain (`#散文`) forms work, because `app.js` runs `decodeURIComponent`. *(A `?tag=` query string does nothing — this was the Candle Blossoms bug fixed 2026-06-06.)*

## Bilingual rendering (`post.js` + CSS)

- Prose is authored as `.pair` blocks: a `.zh` paragraph plus an optional `.en` paragraph.
- **Desktop:** `.zh` / `.en` sit side-by-side in a 50/50 grid. A pair with no `.en` collapses to a single column (via the CSS `:has()` selector).
- **Mobile (≤720px):** Chinese only by default; `post.js` injects a 3-way segmented toggle (中 / 中 + EN / EN) whenever the article contains any `.en`.
- **Verse mode:** add `class="verse"` to `.article-body` → `white-space: pre-line` preserves a poem's line breaks (and full-width-space spacing within a line). *Added 2026-06-06; applied to the Ballad.*
- **Share buttons** (threads / facebook / x / linkedin / copy link) are injected by `post.js` after `.article-tags` on every post — old and new alike, no per-post markup needed.
- **Previous / next nav** (added 2026-06-15) — `post.js` injects chronological neighbour links (older left, newer right) after the share buttons. It reads `POSTS_NAV`, an ordered (newest-first) array of `{slug, title, url}` that `build.js` regenerates between `/* BUILD:NAV_START/END */` markers at the top of `post.js`. Same single-source pattern as the homepage `<noscript>` list: one build step covers every post, no per-post markup, no backfill. Current post is matched by slug from the URL path; first/last posts show only the one neighbour that exists. `build.yml` now commits `post.js` alongside the other generated files.

### In-article image layouts (hand-coded; CSS already supports them)
- Default `<img>` — full column width.
- `<figure>` + `<figcaption>` — captioned image.
- `.img-row` — two images side-by-side (stacks on mobile).
- `.img-inset` — smaller centred image.

These exist in `style.css` and `_TEMPLATE.html` but the **editor cannot produce them** (see below).

## MiniGames (`/minigames/`) — added 2026-06-13

A small section listing browser games built for fun. Two layers:

- **Gallery hub — `minigames/index.html`:** a **standalone static page** (not processed by `build.js`). The "小遊戲 · Mini Games" label sits as **small text under the site identity** in the masthead. - **The "Cats" game — `minigames/catspuzzle/`:** a Star-Battle / Queens-style logic puzzle (one cat per colour/row/column, no two cats touch incl. diagonally). Built with **React + TypeScript + Tailwind 4 + Vite**, fully client-side, themed in Library Linen. Has a "← Mini Games" link (→ `/minigames/`) and a footer link home. **Only the built output is in the repo** (`index.html` + `assets/index-<hash>.js/.css`) — *the React source is not in the repo.*
- **The philosophy quiz — `minigames/philquiz/`:** a 36-question late-night bar chat quiz matching players to 16 thinkers. Built with **React + TypeScript + Vite**, Library Linen themed, bilingual (zh/en). **Deployed like catspuzzle** (`index.html` + `assets/`). Source lives alongside at **`minigames/philquiz/source/`** (`npm run build` writes to the parent folder). Matching is **cosine-based** (`SCORING_VERSION 3`); per-philosopher share landing pages live at `minigames/philquiz/r/<id>/` (og-tag shells, noindex, regenerate with `source/scripts/gen-share-pages.mjs`). **After any change to question weights or centroids, run `node source/scripts/simulate.mjs`** and keep every thinker's persona self-recovery ≥~50% and uniform-random top-1 under ~15%. See `minigames/philquiz/source/README.md`.

### 🎮 The Game Contract (added 2026-06-15)

Games are **apps, not posts**: each game's *interior* (gameplay UI, board, controls,
layout, animation) should differ — **do not template it.** But every game shares an
*outer contract* so it belongs to the site and is discoverable. **Standardise the
shell; keep the interior free.** (Same philosophy as posts having a rigid
`_TEMPLATE.html` head but free body content — just inverted: here the body is free and
only the shell is fixed.)

**FREE — the interior.** Everything about how the game looks and plays. A puzzle, an
arcade game, a word game should look nothing alike. Reference the existing game(s) for
*tone* (quiet, paper, Library Linen restraint), **not** for layout.

**FIXED — the contract every game must honour:**

1. **Frame & tone** — use the Library Linen tokens (below) + the three site fonts;
   restraint over flash. Provide a `← Mini Games` link (→ `/minigames/`) and a link
   home (→ `/`). The game must visually read as part of yykan.uk.
2. **Page meta** — `minigames/<game>/index.html` carries the head block below
   (title / description / canonical / robots / OG / Twitter / JSON-LD). Same discipline
   as `_TEMPLATE.html` for posts.
3. **Registration — 3 synchronized edits (the easy-to-forget part):**
   - `minigames/index.html` → append to the `GAMES` array: `{ title, excerpt, cover, url, tags, meta }`
   - `scripts/build.js` → add to `STATIC_PAGES` (→ sitemap): `{ file, loc, priority }`
   - `scripts/build.js` → add a line to the `## Mini Games` list in the `llms.txt` generator
4. **Share image** — a 1200×630 **PNG** `images/og-<game>.png` (raster; SVG doesn't
   render in social scrapers). Until it exists, the page falls back to `og-minigames.png`.
5. **Hub cover** — an image in `images/` for the gallery card (SVG is fine here —
   same-origin, not a social scraper).
6. **Asset hygiene** — Vite `base: './'` (relative paths, works from the subfolder).
   Prefer **stable output filenames** so updating a game is a plain overwrite; if
   filenames stay hashed, deleting the old hashed files on update is mandatory.

## The editor (`editor.html` + Worker)

- Single-page composer; `noindex` and disallowed in `robots.txt`.
- **Auth:** GitHub OAuth → the Worker (`kanblog-editor-api`) → a session token kept in `localStorage`. The `/auth/start` and `/auth/callback` routes are forwarded to the Worker by `_redirects`.
- Builds the full post HTML client-side (SEO meta, OG, Twitter card, BlogPosting JSON-LD) and POSTs it to the Worker's `/api/commit-file`, which commits to the repo. The commit then flows through the normal build/deploy pipeline.
- **Body composer supports `.pair` blocks only** (Chinese + optional English), plus a single cover-image field with alt text.

**Known limits (relevant to illustrated "上文上圖" posts):**
- Publish-only — it cannot edit an existing post (edit via GitHub directly).
- No inline images, figures, blockquotes, headings, or verse. Any post needing these is hand-coded from `_TEMPLATE.html`.
- *Highest-leverage upgrade:* an "image block" in the composer (src + alt + caption + layout) emitting the HTML the CSS already supports. (See Roadmap.)

## Domain, redirects, Cloudflare

- **Custom domains:** `yykan.uk` (apex, canonical) and `www.yykan.uk` — both active, SSL active. **`www` 301s to the apex** via a Cloudflare **Redirect Rule** named `www to apex` (zone `yykan.uk` → Rules → Redirect Rules; `http.host eq "www.yykan.uk"` → dynamic 301 → `concat("https://yykan.uk", http.request.uri.path)`, preserve query string). Done & verified 2026-06-14 (path + query preserved, no loop). The `www.yykan.uk/* → yykan.uk` line in `_redirects` is a **dormant fallback only** — Cloudflare Pages `_redirects` matches path, not hostname, so it never fires; the Redirect Rule is what does the work (same pattern as the pages.dev Bulk Redirect below).
- **Old domain:** `kanblog.pages.dev` → `yykan.uk` via a Cloudflare **Bulk Redirect** (account-level rule, subpath + query + path-suffix preserved) and the `_redirects` 301.
- **Pages project:** `kanblog` (the source of the old `*.pages.dev` URL). Auto-deploys on every GitHub push.
- **Security headers (`_headers`, 2026-06-14):** Cloudflare Pages serves the repo-root `_headers` file on every response. Sets `Strict-Transport-Security` (1y, `includeSubDomains; preload`), a **tailored CSP** (allows Google Fonts `style-src`/`font-src`, inline JSON-LD via `script-src 'unsafe-inline'`, and the editor's Worker via `connect-src https://kanblog-editor-api.7r6r7hf474.workers.dev`), `X-Frame-Options: DENY`, `Permissions-Policy`, plus a 1-year `Cache-Control` on `/images/*` (previously 4h). ⚠️ **CSP note:** the `connect-src` Worker allowance is load-bearing for the editor — without it `editor.html`'s commit/OAuth `fetch()` would be blocked. If the editor ever stops committing with a console `Refused to connect`, check this directive first. Editor login + commit round-trip **verified working under the CSP (2026-06-14)**.

## Known issues & design notes

- **`SAMPLE_POSTS` naming debt** — the live list is misleadingly named (see "The one coupling that can bite you").
- **Masonry reading order** — CSS columns fill top-to-bottom *per column*, so the visual order does not strictly match newest-first left-to-right. Accepted trade-off for zero JS.
- **Card cover CLS — accepted by design, won't-fix (decided 2026-06-15)** — the masonry deliberately lets each card keep its natural image height (Pinterest-style freeflow), so card covers carry no fixed dimensions and can shift slightly as they load. The "freeflow *and* no-CLS" fix (reserving natural-ratio space from known dimensions) was considered and **declined**: Kaga wants to keep the effortless, automatic Pinterest-style layout. Not a defect — a chosen trade-off.
- **Article hero CLS — won't-fix (decided 2026-06-15)** — the full-width `.article-cover` has no `width`/`height`; same call as above, declined for the same reason.
- **Person entity — topical `knowsAbout` added (2026-06-15); CV-style identity still declined** — the distinction matters: *what the writing is about* is honest metadata that helps AI answer-engines surface the site, so a bilingual `knowsAbout` topic list (移英 / 香港 / 雙語散文 / 遊記 / 哲思 + English mirrors) was added to the Person node in `index.html` (WebSite ▸ author) and `about.html` (AboutPage ▸ mainEntity). *Who the author is* (`jobTitle` / `alumniOf` / credentials / a fabricated CV) stays **declined by design** — 言又勤 is a deliberate persona; the site speaks for itself. So: topics yes, manufactured identity no.
- **Font weight** — Noto Serif HK loads four weights (400/500/600/700); `700` is unused. Note: browsers do not download unused-weight font files, so dropping `700` trims only a little font CSS, not real bytes. A meaningful payload cut means dropping a *used* weight (e.g. `400 + 600`, losing `500`) — a visual trade-off, on the Roadmap.
- **Editor** — publish-only, pairs-only (see editor section). *Open.*
- **build.js meta regex** — attribute-order-sensitive (`name`/`property` before `content`).

*Resolved 2026-06-15:* **About single `<h1>`** (the masthead site-title was demoted to a `<div>` so the page's only `<h1>` is "About"); **About `twitter:card` → `summary_large_image`**; **MiniGames `<noscript>`** static link list added (hand-maintained — add a `<li>` when adding a game); the missing `.site-title-en` / `.site-subtitle` / `.sep` masthead styles; `var(--font-sans)` (undefined) → `var(--font-body)` in `404.html` **and** `minigames/index.html`; a stale `/about.html` footer link in MiniGames → clean `/about`; `editor.html` tag-split separator aligned to `build.js` (`/[,，]/`); `lang="en"` on all `.pair .en` (via `post.js` + inline in `about.html`); dead CSS removed (`.lang-toggle.has-en`, empty `:first-letter`).

*Resolved 2026-06-06:* meta-text contrast (`--ink-faded` → `#75603e`), `prefers-reduced-motion`, `:focus-visible`, site favicon, homepage `og:image`, and the legacy-name strip in `readTitle`.

*Resolved 2026-06-13:* **soft-404** (Cloudflare returned 200 + homepage for any missing URL → fixed with a branded `404.html`, now real 404s); Person `sameAs`; raster OG images (SVG didn't render in social scrapers); `/minigames/` + game in the sitemap; auto-generated `llms.txt`; game-page canonical/OG/Twitter meta; default OG for About + future cover-less posts.

- *Card + article-hero CLS* — **declined 2026-06-15** (keep Pinterest freeflow; see Known issues). Not on the roadmap anymore.
- *Font payload* — only worthwhile if dropping a *used* weight (e.g. `400 + 600`); a visual decision, not free. Dropping the unused `700` alone is near-zero benefit.

**Features:**
- ~~Previous / next ("read next") links~~ — **done 2026-06-15** (chronological, via `post.js` + `POSTS_NAV`; see Bilingual rendering section).

- Dark mode (`prefers-color-scheme`) — fits the candlelight theme.
- Person entity: topical `knowsAbout` **added 2026-06-15**; CV-style identity (`jobTitle`/`alumniOf`) **declined** (let the site speak for itself; see Known issues).
- Client-side search over the post list — when the corpus grows.
- Cloudflare Web Analytics — free, cookie-less.
- Date-indexed archive page — when the corpus grows.
